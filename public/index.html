Introduction

Everything you need to know about this Toys REST API and how does it works

What this API about?

The Toys REST API is a component of a server-side project built using several technologies, including Node.js, Express, axios, bcrypt, cors, dotenv, joi, jsonwebtoken, and mongoose. For more information about the project, please refer to the following link on its GitHub project

The server is responsible for handling requests to the Mongo DB database and can perform various operations on the users and toys routes, including GET, PUT, POST, and DELETE requests.

User Interface

Layout, navigation, and functionality of the user interface, including how to access various features and functions

Options of UI interaction

The user interface for this application can be accessed through this web URL input directly in your web browser or through an API platform like Postman, wtih all request to the same URL. For the purposes of this documentation, we will use Postman to provide clear instructions. Please note that this server has been tested and works best with hand requests or requests made through Postman, and may not function well with other programs.

The user interface is structured around two main routes: toys and users. To use the application, users must first register by making a POST request, which will be explained in detail later. Once registered, users can create new toys in the toys route and delete any items they have created.

Without registering, users can still access the toys route, but will only be able to make GET requests to retrieve information about toys and sort/search by name or category.

User Roles and Permissions

Different roles and access levels for users, and what actions they are authorized to perform.

User and Admin roles

The Toys API has two levels of user roles and permissions. By default, all users are considered regular users and can only perform limited actions. Regular users can delete only the items they have created and modify their password, email, or name. Only an admin can grant additional permissions or elevate a user to the second user role.

The second user role is called an admin, and can only be granted directly in the Mongo database. Admins have full control over the users and toys routes, allowing them to create, update, and delete items without any limitations. They can also view all users and their created items. Due to their elevated permissions, admin accounts must be carefully managed and restricted to trusted individuals. At this moment its not possible to require change status to admin account

Functionality

Features and functions of the application, including how to use each feature and what it does.

Implemented libraries, techniques and pacckages

The server was created using various npm packages. One of the main packages used was Node.js runtime, which is essential for developing server-side features based on its syntax.

  • Express was used to create the server and create routers for handling all requests. This package is a fast, unopinionated, and minimalist web framework for Node.js that provides various features to build robust and scalable web applications.
  • Mongoose was used to establish a connection between the server and MongoDB. It provides a schema-based solution to model your application data and supports validation, query building, and data casting. Additionally, Mongoose also enables the creation of specific required functions, such as creating basic schema models.
  • Joi was used to verify models before sending them to the database to ensure that they contain the required values and keys. This package is a powerful schema description language and data validator for JavaScript.
  • Axios was used to handle asynchronous requests to the database and get responses. This package provides a simple and easy-to-use API for sending HTTP requests and handling responses asynchronously.
  • jsonwebtoken was used to create and pass tokens to recognize user actions and logins. It is a package for JSON Web Tokens (JWT) that can be used to create and verify tokens.
  • Cors was used to turn off IP verification and disable strict mode of access to the server. This package is a Node.js package for providing a Connect/Express middleware that can be used to enable Cross-Origin Resource Sharing (CORS) for the web application.
  • Bcrypt was used to hash the password provided by the user. It is a password-hashing function designed to be slow and computationally expensive to make it difficult to perform brute-force attacks against user passwords.

Routes

Different routes or endpoints available in the application's API, what they do, and how to use them.

Description

This API consists of two main routes, /toys and /users, each serving different functionalities. Additionally, there is a /main endpoint that will guide you on what to do in case you are lost. Finally, if user enter a non-existing endpoint, the server will return a default 404 error response router.

The toys router

The toys router is responsible for handling requests related to toys. It allows users to perform GET, PUT, POST, and DELETE requests for toys. Additionally, users can sort and search for toys by name, category, and other information. The toys router also handles requests for creating and updating toy information, such as name, description, and price.

The users router

The users router, on the other hand, handles requests related to users. It allows users to perform GET, PUT, POST, and DELETE requests for user data. Users can also sort and search for users by name and other information. This router is responsible for managing user authentication, including user registration, password reset, and user role management.

The main router

The main router is a default endpoint for the API. It's just shows information and recommends to visit pyblic page woth documentation

404 router

The 404 router is an additional router in the Toys REST API that handles any requests for non-existing endpoints. If a user enters a non-existing endpoint, the server will return a default response with a 404 error message. The purpose of this router is to provide a clear response to users and help them understand what went wrong in their request.
